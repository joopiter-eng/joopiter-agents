Summary: Keep the earlier stream close for the active-viewer UX, but stop treating "stream ended" as "response is ready." Gate background completion notifications on the assistant message being durably persisted, and add a small recovery path for chats opened during the post-stream persistence gap so they never show a blank finished state.

Context: `apps/web/app/workflows/chat.ts` now clears `activeStreamId` and closes the stream before `persistAssistantMessage()` runs in `apps/web/app/workflows/chat-post-finish.ts`. That fixed the live-view issue, but it created a semantic gap: `apps/web/hooks/use-sessions.ts` stops polling as soon as `hasStreaming` drops, and `apps/web/hooks/use-background-chat-notifications.tsx` currently treats that transition as completion even though the assistant row may not exist yet. If the user clicks that notification, `apps/web/app/sessions/[sessionId]/chats/[chatId]/page.tsx` renders only persisted DB messages, passes `initialChatActiveStreamId` into `useSessionChatRuntime`, and `apps/web/app/sessions/[sessionId]/chats/[chatId]/hooks/use-session-chat-runtime.ts` only resumes when that ID is still present. After the early clear, there is no stream to resume and no persisted assistant message yet, so the page can look blank and already-finished.

Approach: Use persisted assistant activity, not `activeStreamId`, as the notification-ready signal. Expose the latest persisted assistant timestamp on session summaries, keep polling briefly for sessions that just stopped streaming until that timestamp advances, and only then toast. Separately, when a chat page loads with a trailing persisted user message but no persisted assistant yet, show a compact "finalizing response" state and poll a tiny messages endpoint until the assistant message lands, then hydrate the client messages from that persisted snapshot.

Changes:
- `apps/web/lib/db/sessions.ts` - extend the session sidebar query/result shape with a persisted-assistant timestamp (for example the latest `lastAssistantMessageAt` for the session or latest chat) so the client can tell "assistant saved" apart from "stream just closed." Keep `hasStreaming` as the live-run signal; do not change the earlier close behavior in `chat.ts`.
- `apps/web/hooks/use-sessions.ts` - carry the new persisted-assistant timestamp through `SessionWithUnread`, clone/merge helpers, and returned session data so notification logic can compare previous vs current persisted state.
- `apps/web/hooks/use-background-chat-notifications.tsx` - replace the immediate streaming→idle toast with a two-phase flow: when a session stops streaming, store it in a pending set with its previous persisted-assistant timestamp; keep refreshing sessions while any pending item exists; only fire the toast once the persisted-assistant timestamp advances. Add a timeout so the hook does not poll forever if persistence fails.
- `apps/web/app/sessions/sessions-route-shell.tsx` - pass `refreshSessions` into the background-notification hook so it can keep polling during the short post-stream persistence window even after `hasStreaming` is already false.
- `apps/web/app/sessions/[sessionId]/chats/[chatId]/page.tsx` - derive an `isAwaitingPersistedAssistant` flag from the SSR snapshot (latest persisted message is `user`, `chat.lastAssistantMessageAt` is missing/stale relative to that user turn, and there is no active stream to resume) and pass it to the client chat UI.
- `apps/web/app/sessions/[sessionId]/chats/[chatId]/hooks/use-persisted-response-recovery.ts` - add a focused client hook that polls for the persisted chat snapshot only while `isAwaitingPersistedAssistant` is true. When the assistant message appears, update the in-memory chat messages and stop polling; if the timeout expires, stop and leave a non-blocking fallback state instead of spinning forever.
- `apps/web/app/api/sessions/[sessionId]/chats/[chatId]/messages/route.ts` - add a thin authenticated GET endpoint that returns the persisted chat messages (and any tiny status fields the recovery hook needs). Keep it read-only and scoped to the owned session/chat.
- `apps/web/app/sessions/[sessionId]/chats/[chatId]/session-chat-content.tsx` - render a subtle "Finalizing response…" placeholder during the recovery window and wire in the new recovery hook without mixing transport/persistence polling into the already-large main component.
- `apps/web/hooks/use-background-chat-notifications.test.ts` - cover the new pending-notification behavior: no toast when streaming stops but the persisted timestamp is unchanged, toast only after the persisted timestamp advances, and timeout cleanup.
- `apps/web/app/api/sessions/[sessionId]/chats/[chatId]/messages/route.test.ts` - cover auth/ownership, no-new-message snapshots, and the persisted-assistant-arrived case.
- `apps/web/app/sessions/[sessionId]/chats/[chatId]/page.test.ts` or a focused recovery-hook test - lock in the SSR detection for the post-stream persistence gap and the client recovery behavior.
